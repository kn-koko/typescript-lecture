「まだスラスラ書けない」は順調です

今回は「詰まった」よりも、「調べながらなら書ける」「形が覚えられない」「書く順番が身についていない」という声が中心でした。まとめてコメントします。

調べながら書いている——それが正しい書き方です

「スラスラ書くことはできないけど、調べながら落とし込んでいる」という声がありました。それは現役のエンジニアの書き方そのものです。構文を暗記してから書く人はいなくて、みんな調べながら・補完に助けられながら書いています。調べて手元のコードに落とし込めているなら、今回の内容はもう「使えて」います。

「カスタムエラーの定番形（constructor ＋ super(message) ＋ this.name）は何度か書かないと覚えられなそう」という声も、その通りで大丈夫です。これは書いた回数で手に馴染むタイプの構文なので、当面は教材 8-c の丸写しから始めてください。

覚えるのは「形」より「使いどころ」

細かい書き方は教材を見ればいいので、頭に残すのは次の3つだけで十分です。

  • 落ちるかもしれない処理は try で囲んで catch で受け止める（8-a）
  • 失敗が確定したら throw で早めに脱出する——ガード節（8-b・8-d）
  • エラーの種類を見分けたくなったら、カスタムエラー ＋ instanceof（8-c）

「処理のフローはイメージできるようになってきた」という声がありましたが、まさにそのフローのイメージこそが今回持ち帰ってほしいものです。書く順番は、イメージがあれば後から必ずついてきます。

Q1Error クラスは定義していないのに、いきなり出てきて戸惑った

複数の方から同じ質問がありました。「Error というクラスは標準装備されているクラスという理解であっていますか？」——その理解で正解です。Error は JavaScript が言語として最初から用意しているクラスで、自分で定義したり import したりしなくても使えます。実はこれまでも同じ「標準装備」をいくつも使ってきました。

「例外がクラスで表されるのがしっくりこない。今まで習っていたクラスが大枠で、『エラーを表現するための設計図』を例外クラスと呼んでいるイメージであっていますか？」——このイメージも正確です。クラス＝設計図という理解はそのままでよくて、Error は「エラーという出来事」を表現するための設計図です。new Error("メッセージ") は、その設計図から「今回起きたエラー1件分」の実体（インスタンス）を作っています。メッセージや種類の情報をひとつの荷物に詰めて、throw で投げて catch まで運ぶ——そのための入れ物と考えてください。

そしてカスタムエラー（8-c）は、この標準の Error を extends した子クラスです。Character を継承して Warrior や Priest という種類を作ったのと、まったく同じ構図です。

Q2try-catch と throw new Error は、それぞれどういう役割？

「throw new Error は自分でも想定されるエラーを先出ししておくイメージ？」——かなり良い線です。整理すると、この2つは「投げる側」と「受け止める側」のペアです。attack する側がいれば takeDamage される側がいる、という前回の話と同じ構図です。

  • throw ＝ 投げる側：処理の奥で「これは失敗が確定した」と分かった瞬間に、エラーを報告して脱出する（divide の 0 割り、registerUser の空文字チェック）
  • try-catch ＝ 受け止める側：報告されたエラーを受け止めて、「ユーザーにどう伝えるか」「次にどうするか」を決める

「try-catch だけだと何がエラーか分からないから、それを伝えるために throw new Error が必要？」——ここだけ少し修正させてください。try-catch は、自分が throw を書かなくても働きます。8-a で JSON.parse の例外を受け止めたとき、throw は一行も書いていませんでした。あれは JSON.parse が内部で throw してくれていたのです。つまり throw は「標準の関数やライブラリが中でやっていることを、自分のロジックでもやる」ための道具です。

そして「何のエラーかを伝える」役割を担うのは、throw そのものではなくカスタムエラー（8-c）です。8-d の実例がこの全部の組み合わせで、奥のロジック（registerUser）が ValidationError を throw し、画面に近い側（onSubmit）が catch して instanceof で見分けています。

Q3「こうなるだろう」とエラーを予測しながら書くのは慣れ？ 実際どんなエラーが想定される？

半分は慣れですが、もう半分は定石です。エラーをゼロから予知する必要はなくて、例外が起きやすい場所は実は決まっています。

  • 外から入ってくるデータ：ユーザーの入力（空文字・数値じゃない文字列）、JSON.parse に渡す文字列、ファイルや通信で受け取るデータ
  • 計算の特殊ケース：0 で割る、マイナスや範囲外の値

定石をひとことで言うと「外から来るものは信用しない」です。境界（入力を受け取る場所）にガード節を置く、と覚えておけば、予知能力は要りません。

「いま学習しているコードで実際どんなエラーが想定されるのか、考えても分からなかった」という声もありましたが、これは正しい観察です。いま書いている戦闘シミュレーションは、入力が全部自分のコードの中で決まっているので、例外処理の出番がほとんどありません。例外処理が本領を発揮するのは、画面入力・通信・ファイルといった「外部」とつながってからです。今の段階では「そういう備え方がある」と知っていれば十分です。

Q4try を書くとき、catch と finally は必ずセットで書く必要がある？

セットである必要はありません。ルールは「try 単独は書けない。catch か finally の少なくともどちらか1つが必要」です。組み合わせは3パターンあります。

finally は「成功しても失敗しても必ずやりたい後片付け」があるときだけ足せば十分で、普段の第一候補は try-catch だけの形です。

Q5頭の中で「ざっくりこんな感じかな？」という理解のままで、一旦大丈夫？

一旦それで大丈夫です。今回の到達目標は、次の3つのイメージが持てていることです。

  • 例外が起きると、プログラムはその場で止まってしまう
  • 落ちそうな処理は try-catch で受け止められる
  • 失敗が確定したら throw で早めに脱出できる

例外処理は「実際に必要になる場面で使って身につく」タイプの内容なので、いま完璧に書ける必要はありません。練習問題で「ほぼ詰まった」という声もありましたが、どこで詰まったかを次のアンケートや授業でそのまま教えてもらえれば、個別にフォローしてこのページでも解説します。
