つまずき解説

練習問題でつまずきが多かったところ

今回いちばん多かったのは「蘇生（revive）をどう実装するか」でした。関連するつまずきをまとめて解説します。

「蘇生」はどう実装する？ — 特別な仕組みは要らない

「倒れている」とは hp が 0 になった状態のことです。つまり蘇生＝倒れているキャラの HP を回復させることなので、蘇生専用の新しい仕組みは必要ありません。isDead() で倒れているかを確認してから、いつもの takeHeal で回復させれば、それがそのまま蘇生になります。

回復量は「現在のHP＋数値」で計算するべき？

その必要はありません。呼ぶ側は「いくら回復させるか」だけを渡し、HP を実際に増やす計算は、回復される側のキャラクター自身（takeHeal の中）に任せます。attack と takeDamage の関係とまったく同じで、これがカプセル化の考え方です。

最大HPを超えて回復してしまう

上限の管理も takeHeal の中でやるのがきれいです。Math.min を使うと1行で書けます（if 文で書いても、もちろんOKです）。

「〇〇される側」のメソッドが必要なことに気づけなかった

新しい能力を作るときは、「する側」と「される側」をペアで考える癖をつけると設計しやすくなります。attack する側がいれば takeDamage される側がいる——これと同じで、heal には takeHeal がペアになります。「誰が何をして、誰が何をされるか」を先に書き出してからコードにするのがおすすめです。

「AがBを蘇生した」とログに出したい

name は protected なので、他のキャラクターから直接読むことはできません。そこで getName() のような取得用メソッド（getter）を作って読む——まさに前回やったアクセス修飾子の実践です。ここに自力でたどり着けたなら、前回の内容がしっかり身についています。

implements のエラーが解決できない

よく出るのは Class 'HolyPriest' incorrectly implements interface 'Healable' のようなエラーです。これは「interface で約束したメソッドがまだ足りない、または引数・戻り値の型が合っていない」という意味です。interface に書いたメソッドをすべて、同じ形で実装できているか確認してみてください。解消できなかった部分は、次回の授業でも扱います。

Q & A

いただいた質問への回答

アンケートでいただいた質問に、まとめて回答します。

Q1interface はポケモンでいう「技」のようなイメージ？

とても良い線です。より正確に言うと、interface は技そのものではなく「この技が使えます」という約束（宣言）です。技の中身（実装）は、それぞれのクラスが自分で書きます。

Healable を implements した Priest と HolyPriest は、どちらも必ず heal を持っていることが保証されますが、回復量や動きはクラスごとに違っていて構いません。「技の名前と使い方だけ決まっていて、実際の効果は使い手ごとに実装する」——このイメージで大丈夫です。

補足
RPGに馴染みがない方へ：例え話はポケモンでも何でもOKです。自分のよく知っている世界に置き換えて考えられるのは、むしろ理解が進んでいる証拠なので、どんどんやってください。

Q2キャラクターを増やすときは、1キャラ1ファイル？ それとも new で増やす？

「サトシのピカチュウとか、野生のピカチュウとか、ピカチュウはいっぱいいるじゃないですか」——素晴らしい質問で、これはクラスとインスタンスの違いそのものです。

  • クラス＝種類の設計図：「ピカチュウ」という種族
  • インスタンス＝個体：サトシのピカチュウ、野生のピカチュウ

なので、両方を使い分けます。新しい種類（リザードンなど）を追加するときはクラスを作り（1クラス1ファイルにするのはこの講座の慣習です）、同じ種類の個体を増やすときは new でインスタンスを作ります。個体ごとの違いはコンストラクタの引数で表現します。

サトシのピカチュウと野生のピカチュウは「同じ Pikachu クラスから作った、別のインスタンス」です。授業で new Priest("Alice", 100) と名前を渡していたのも同じ理屈です。

Q3前回の課題で、Enemy クラスを継承して新しい敵を作ったのは合っている？

合っているかどうかは「その敵が何をするか」で決まります。判断の目安は次の通りです。

  • 強さ（HP・攻撃力など数値）が違うだけ → 同じクラスから new で個体を作れば十分
  • 行動（メソッドの中身や独自の能力）が違う → 継承して override するのが活きる

どちらで書いても、動いていれば「間違い」ではありません。継承を選んだこと自体は、行動の違う敵を作るための正しい選択肢のひとつです。

Q4interface はどんな場面で使う？ もう一度説明してほしい

次回の授業で改めて復習します。先にポイントだけまとめると、interface が向いているのは「一部のクラスにだけ、共通の能力を横断的に持たせたいとき」です。

  • 全員が持つもの（attack、hp）→ 親クラス Character に
  • 特定の職業だけが持つ能力（heal、revive）→ interface に

継承（extends）は親を1つしか選べませんが、implements は複数できます。だから HolyPriest のように「回復もできて、蘇生もできる」を表現できる——ここが interface の強みです。詳しくは教材本編の 7-d も参照してください。

Q5細かい記述のルールが覚えきれない

暗記は不要です。押さえるのは class HolyPriest extends Character implements Healable, Reviveable という形だけで、細部はエディタの補完とエラーメッセージが教えてくれます。教材ページをリファレンスとして手元に置いて、書きながら慣れていけば十分です。

Q6AIツールの利用について（オプトアウトの設定）

方針は「入力した内容が学習データとして使われない設定（オプトアウト）を有効にした上で利用する」ことです。個別のツールや設定がこの条件を満たしているかは製品ごとに異なり判断が難しいため、ICTS の案内・窓口で確認してください。
