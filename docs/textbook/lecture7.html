Chapter 7-a

interface の基礎

interface とは何か。まず概要から。

interface は、いちばん分かりやすく言うとクラスの「型だけ」バージョンです。重要なのは、実装を持てないこと。下の例では、name が string 型、age が number 型、greet が string を返すメソッド──という型の情報だけを定義していて、値や処理の中身は書きません。

書き方はクラスにかなり似ています（name/age はプロパティ的、greet はメソッド的）。使い道は2つ。1つは純粋に型の定義として。もう1つはクラスとセットで使う方法です。

POINT
interface ＝ クラスの型だけ版。実装は持たない、型の情報だけ。使い道は「型定義として」と「クラスに implements させて」の2つ。
Chapter 7-b

型定義としての interface

1つ目の使い方。type とほぼ同じ。

以前、人を表すオブジェクトの型を type Person = { ... } で定義したことがありました。この type で定義していた型は、そのまま interface に置き換えられます。type を interface に変え、= を消すだけ。

どちらを使っても、このケースではまったく同じものとして使えます。型が合わなければ同じようにエラーになります。詳しく見ると型の合成・継承などで違いはありますが、今の段階ではどちらでも構いません。ただ、プロジェクト内では type か interface に統一しておくと読みやすいです。

POINT
オブジェクトの型定義としては type とほぼ同じ。type X = {...} ⇔ interface X {...}（= の有無）。どちらかに統一して使う。
Chapter 7-c

クラスと interface（implements）

2つ目の使い方。クラスに「この能力を持つ」と約束させる。

Character を継承した新しいキャラクター Priest（僧侶）を追加します。ただ Priest は攻撃役ではないので、attack は「攻撃できない」にしておき、代わりに回復メソッド heal を持たせたい。

この「heal できる」という性質を、interface Healable として定義します。「heal(target) というメソッドを持つ」という契約です。そして Priest が implements Healable と書くことで、「私は Healable です（heal を実装します）」と約束します。実装しなければエラーになる点は、abstract メソッドと似ています。

回復は、対象の HP を増やす処理です。ただ hp は protected なので、他人の HP を直接いじれません。そこで Character に takeHeal メソッドを足し、メソッド経由で HP を増やします（ダメージの takeDamage と同じ考え方＝カプセル化）。

Priest も Magician も Character を extends しつつ、Healable を implements しています。どちらも「キャラクターの一種」であり、かつ「回復できる」。この2つを別々に表現できるのがポイントです。

POINT
extends は「〜の一種である」（親クラスを継承）、implements は「〜できる」という性質を約束する。interface が定めたメソッドは必ず実装する契約になる。
Chapter 7-d

interface と継承の違い

なぜ継承だけでは足りず、interface が要るのか。

interface の強みがいちばん分かりやすいのがここです。継承（extends）は、親を1つしか持てません（単一継承）。だから「特定のクラスにだけ、このメソッドを持たせる」という枝分かれは、継承だけでは表現できません。

たとえば「回復できる（Healable）」のは Priest と Magician だけ、「上級攻撃ができる（SuperJob）」のは SuperPriest だけ──こういう横断的な能力は、interface を必要なクラスにだけ implements することで表現できます（1つのクラスが複数の interface を持てる）。

使い分けの指針はこうです。継承するすべてのクラスが絶対に持つメソッド（attack など）は、親クラス（Character）に書く。一方、特定の職業だけが持つメソッド（heal、superAttack など）は、interface で定義して、その属性を持つクラスにだけ付ける。

継承（extends）interface（implements）
意味「〜の一種である」「〜できる」という性質
数親は1つだけ（単一）複数付けられる
実装親の実装を引き継ぐ実装は持たない（契約のみ）
使いどころ全クラス共通のメソッド（attack）特定クラスだけの能力（heal）
POINT
extends＝縦の系譜（1つの親）／implements＝横断的な性質（複数付与可）。共通のメソッドは親クラスにまとめ、限定的な能力は interface で必要なクラスにだけ付ける。これが継承と interface の使い分け。
演習

演習：「蘇生できる」を interface で表現する

複数の interface を implements し、限定的な能力を横断的に付ける——7-d の使い分けを手を動かして確かめます。

演習 1 ・ Reviveable インターフェースと HolyPriest クラス

問題

  • 「蘇生できる」を表す Reviveable インターフェースを作る（revive(target: Character): void）。
  • 上位僧侶 HolyPriest クラスを作る：Character を extends しつつ、Healable と Reviveable の両方を implements。
    • attack：攻撃できない旨をログに出す
    • heal(target)：target.takeHeal(15)
    • revive(target)：target.isDead() なら大きく回復させて蘇生、生きていれば「まだ倒れていない」
  • main.ts で、倒れた味方を HolyPriest が蘇生できることを確認する。

演習 2 ・ Healable[] でまとめて回復（発展）

問題（授業の課題には含まれない追加問題です）

  • Healable を実装するクラス（Priest・HolyPriest）のインスタンスを Healable[] の配列にまとめ、全員で一斉にヒールする。
POINT
クラスは違っても Healable という共通の契約を持つので、Healable[] にまとめて同じように扱える（多態）。つまずきやすいポイントの解説は第7回アンケート回答にまとめています。
