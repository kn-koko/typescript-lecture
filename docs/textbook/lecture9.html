Chapter 9-a

Map — キーと値のペア

オブジェクトと似ているが、キーを「後から・動的に」決められるのが違い。

まず「組み込みAPI」とは何か。TypeScript に最初から用意されている（組み込まれている）クラスのことです。よく使う処理や「こういうデータの持ち方をしたら便利ですよね」というものがあらかじめ用意されていて、これをうまく使ってコードを書いてくださいね、というのが組み込みAPIです。必ず使わないといけないものではありませんが、知っておくとより良いコードが書けます。今回は Map・Set・Date・RegExp の4つ。この4つは繋がっているわけではなく全然別のことをやるものなので、1個1個独立したものとして受け取ってください。

1つ目の Map はクラスなので、new Map() という形でインスタンスとして宣言します。後ろの <string, number> は「キーが string、値が number のペアを格納します」という宣言です。ここでは userAges——ユーザーの年齢を格納する Map で、「Alice は 25 ですよ」「Bob は 18 ですよ」という対応を入れていきます。イメージとしては { Alice: 25, Bob: 18 } というオブジェクトを持っている感じです。

ではオブジェクトと何が違うのか。オブジェクトのキー（name や age のような）はコードにベタ書きされた値で、あらかじめ決めておかないといけません。でも「Alice」「Bob」のような名前をキーにしたい場合、どういう値が来るか予想できないので、最初にコードに書いておくのが難しい。そういうときに使えるのが Map です。Map は「キーは string、値は number」ということしか決まっていないので、実際の対応関係は後から入れられるわけです。

値を入れるときは set、取ってくるときは get でキーを指定します。has は「このキーはありますか？」の確認で、has("Bob") は true、存在しない "Charlie" なら false。delete でキーごと消すこともできます（動画では delete("Bob") で has が false になるのを確認してから、次のループに Bob を残すためコメントアウトしています）。

Map は複数のペアを持てるので、for 文でループを回せます。ただし書き方がちょっと特殊です。for (const [name, age] of userAges)——この大括弧で、キーと値をそれぞれの変数に拾ってくることができます。イメージとしては、1周目は [Alice, 25]、2周目は [Bob, 18] が配列みたいな形で返ってきて、先頭を name、後ろを age に入れる、という感じです。この書き方は特殊なので覚えておいてください。

POINT
  • 組み込みAPI＝最初から用意されているクラス。new して使うのは自分で作ったクラスと同じ。
  • Map はキーと値のペアを複数格納する。set / get / has / delete。
  • オブジェクトと違い、キーをプログラムの中にベタ書きせず動的に指定できるのが便利。
  • ループは for (const [name, age] of map) の形（キーと値を同時に受け取る特殊な書き方）。
Chapter 9-b

Set — 重複を許さない配列

同じ値をもう一度入れても、何も起こらない。それが特徴であり武器。

2つ目の Set は、一言で言うと「重複を許さない配列」です。new Set<number>() で「number 型の要素が入る集合」を宣言し、add というメソッドで値を追加していきます。1、2、3 と追加すると、どんどん後ろに要素が足されていって——このへんは配列っぽいですね。

では配列と何が違うのか。同じ値をもう一度 add しても、何も起こらないんです。もう一度 add(2) しても「1, 2, 3, 2」とはならず「1, 2, 3」のまま。つまり Set で表されているデータの集合は、中身が絶対に重複していないことが保証されているわけです。

Map と同じく has で「この値はありますか？」を確認できます。もう1つ、size というプロパティで中に何個入っているかが分かります（length みたいなイメージです）。

この size が効いてくる使いどころを1つ。たとえば施設の「今月の予約者名簿」がバーッとあって、同じ人が何回も予約しているかもしれない中で「結局、何人が予約したの？」を知りたいとします。配列を1個1個回して重複チェックすると結構大変ですが、配列を全部 Set にぶち込んでしまえば、TypeScript 側が勝手に重複を排除してくれる。あとは size を見るだけで人数が分かります。頭をひねってうまく使うと処理がかなり簡単に書けて、ループを消せたりもするので、覚えておくと役に立ちます。私は結構 Set が好きです。

ループは Map と違って特殊な書き方は必要なく、配列と全く同じ素直な形で回せます。

POINT
  • Set は重複を許さない集合。同じ値を add しても何も起こらない。
  • has で存在確認、size（プロパティ）で個数。
  • 「配列を Set にぶち込む」だけで重複排除が終わる——重複を数えたい・除きたいときの第一候補。
Chapter 9-c

Date — 日付と時刻

現在時刻の取得と日付の計算。ただし「タイムゾーン」に癖がある。

3つ目の Date は、読んで字のごとく日付や時刻を表すインスタンスです。new Date() と書くと、現在時刻のインスタンスが作られます。中身を確認するには toISOString() を使います。ISO 形式というのは、日付や時刻の表記が国によってバラバラなので、統一された形式として決められているものです。

ここで出力を見ると——時刻が9時間ずれています。収録時は7月23日の0時14分でしたが、出力は「7月22日の15時14分」。実は new Date() で作られたインスタンスは、世界標準時（UTC）を基準にした値を持っていて、toISOString() はその UTC の時刻を出します。日本は9時間の時差があるので、日本から見ると9時間巻き戻った時刻が出てくるわけです。

プログラムの中で時刻を扱うときは、基本的に世界標準時を基準に考えます。国によって時間そのものがバラバラなので、統一的に扱うにはどこかに基準が必要——じゃあ標準時にしましょう、というわけです。日付の計算は UTC で行い、表示するときにその国の形式に直す。日本時間で出すには toLocaleString("ja-JP") を使います。この「どこの時間で表示するか」をタイムゾーンと言い、ちゃんと指定しないと全然違う時刻が出てしまうので、時刻を扱うアプリケーションを作るときはかなり気を使う必要があります。最終課題でタイマーなどを作る方は、このタイムゾーン問題に注意してください。

一部だけ取り出すメソッドもあります。年だけなら getFullYear()。月は getMonth() ですが——そのまま出すと「7月なのに 6」と出ます。実は1月が 0 で、配列と同じ「最初は0番目」方式なので1個ずれているんです。実際の月が欲しければ + 1 してあげる必要があります。日は getDate() で、こちらは 1 は足さなくてよくて、そのまま 23 と出てくれます。

日付の計算もやってみます。setDate(now.getDate() + 7)——今日が23日なので「23 + 7 = 30」を日付にセットする、つまり日付の部分だけを書き換えて1週間後に進める書き方です。時刻や年月はそのままで、日付だけが 30 になります。同じように setMonth() などで月だけを書き換えることもできます（1ずれるのは get と同じで、コードの setMonth(3) は4月です）。ちなみに動画では setMonth(1)＝2月を試したところ、その時点の日付が30日で「2月30日」は存在しないため、自動で3月2日に繰り越されるという一幕もありました。存在しない日付は隣の月に押し出されます。

POINT
  • new Date() は現在時刻。中身は UTC（世界標準時）基準——toISOString() は日本時間より9時間前に見える。
  • 内部の処理は UTC、表示は toLocaleString("ja-JP") が定石。タイムゾーンの指定を忘れると全然違う時刻が出る。
  • getMonth() は0始まり（1月=0）。実際の月は + 1。
  • setDate / setMonth で一部だけ書き換えて日付計算。存在しない日付は隣の月に繰り越される。
Chapter 9-d

RegExp — 正規表現

文字列の「パターン」を検出し、置き換える。ただし深入り注意。

最後の RegExp は Regular Expression の略で、正規表現を表すものです。何のために使うかというと、文字列の中から特定のパターンを検出する、もしくは抜き出して置き換えるためのものです。たとえば「メールアドレスを表す正規表現」——アットマークの前に何らかの文字列があって、その後ろに文字列とピリオドが続いて……というパターンを、正規表現という記法で表現できます。

ただ、先に言っておくと、今回の4つの中でいちばん深入りするとやばいのがこの正規表現です。試しに「メールアドレス 正規表現」で検索してみると、とても覚えられない長さのものが出てきます。笑っちゃうんですけど、私も覚えていません。今は AI に「こういう正規表現が欲しい」と言えば出してくれますから、書き方を暗記する必要は全くありません。「文字列のパターンを扱うときは正規表現が使える」と頭に入れておけば、基本的にはそれで十分です（じっくり見ると実はそんなに難しいことは書いていないので、興味があれば AI と話しながら読んでみると面白いですよ）。

実際に使ってみます。書き方が特殊で、スラッシュで囲みます。/hello/i——最後の i は「大文字小文字を区別しない」というフラグです。なのでこのパターンは hello も Hello も hElLo も検知できます。pattern.test(文字列) で、その文字列にパターンが含まれるかどうかを検証できます。"Hello, world" は true、"goodbye" は false ですね。

検出だけでなく置き換えもできます。今度はパターンからではなく、置き換えたい文字列の方から text.replace(検出するもの, 置き換え後) と呼び出します。実は replace 自体は正規表現なしでも text.replace("Typescript", "JavaScript") のように普通の文字列で使えるのですが、検出する側を正規表現にすると大文字小文字を無視して検出・置換できます。コードの /typescript/i は小文字で書いていますが、i フラグのおかげで本文中の Typescript を検出して置き換えてくれます。ここが普通の文字列 "typescript" だったら、本文に小文字の typescript は存在しないので置き換わりません。

最後に1つ。Map も Set も Date も new Map() のように書いてきたのに、この pattern はどこにも new と書いていません。実はこの /hello/i というリテラル自体が RegExp クラスのインスタンスになっています。正規表現だけちょっと特殊で、new RegExp(...) と書かなくてもインスタンスが作られる仕様なんです。つまり Map・Set・Date、それに前回の Error も含めて、組み込みAPI は「クラスのインスタンスを作って使う」仲間だということです。

POINT
  • 正規表現＝文字列のパターンを定義して、test で検出、replace で置換。
  • /hello/i のようにスラッシュで囲む。i フラグで大文字小文字を無視。
  • 複雑なパターンの書き方は暗記不要——調べる・AIに出してもらうでOK。「パターンには正規表現」とだけ覚える。
  • /…/ リテラルは RegExp クラスのインスタンス。組み込みAPIはどれも「クラスを使う」仲間。
