Chapter 8-a

try-catch の基礎

例外が起きるとプログラムは止まる。それを受け止める最初の構文。

まず「例外」とは何か。下のように、JSON の形式に従っていない文字列を JSON.parse に渡すと——コードを書いた時点ではエラーは出ませんが、実行するとその行で例外が発生します。JSON.parse は JSON 形式の文字列を JavaScript のオブジェクトに変換する関数で、渡した文字列が不正だと「JSON として適切じゃない」と例外を投げるのです。

やっかいなのは、この手のエラーは発生した所でプログラムを強制終了させること。つまり、後ろに書いた console.log はもう実行されません。

これでは困る。そこで try-catch です。落ちる可能性のある処理を try { } で囲み、例外が起きたときにやりたいことを catch (error) { } に書きます。try の中で例外が発生すると、その先の行は飛ばして catch に移り、そこで受け止める。しかも——例外でクラッシュするのではなく、catch を抜けて次の処理へ普通に進めます。

実行すると出力はこの順番になります。try の中で例外が起きても、プログラムは止まらずに最後まで走りきるのがポイントです。

POINT
例外は、発生した所でプログラムを止めてしまう。落ちそうな処理を try で囲み、catch (error) で受け止めれば、クラッシュを防いで後続の処理に進める。これが例外処理の一番基本の形。
Chapter 8-b

throw と finally

自分から例外を投げる。そして「必ず実行する」処理。

電卓の divide（割り算）を思い出してください。割り算では 0 で割るのを防がないといけません（0 で割ると Infinity が返り、その値で計算を続けると後続がどんどん壊れていく）。以前はこれを return false や number | boolean で表現しましたが、実はあまり良くない。もっと良い方法が例外です。例外には「後続の処理を問答無用でキャンセルする」力があるからです。

b が 0 のときは throw new Error("...") で例外を投げます。throw は「例外を投げる」という意味で、基本は Error クラスのインスタンス（new Error(メッセージ)）を投げる、と形で覚えてください。こうすれば 0 で割る計算には絶対に進みません。

受け取る catch 側では、error.message でメッセージを取り出せます。ただし catch の error は unknown 型——「まだ何型か分からない」状態なので、そのままでは message を持つとは限らずエラー（赤波線）になります。今の段階では (error as Error) と型を指定して回避します（詳しい書き方は後の回で）。表示は console.error を使うと ⚠️ 付きでエラーだと分かりやすくなります。

そして try-catch の後ろには finally を足せます。finally は、try の中で例外が起きても起きなくても——成功しても失敗しても必ず実行される処理です。上の例は divide(10, 2) が成功するので catch には入りませんが、「この処理は絶対に実行される」は必ず出力されます。失敗した場合も、catch を通ったあと最後に finally を必ず通ります。

POINT
  • throw new Error(...) で、自分から例外を投げて後続を止められる（ガード）。
  • catch の error は unknown 型（今は (error as Error) で回避）。
  • finally は成功・失敗に関わらず必ず実行される。
Chapter 8-c

カスタムエラーと分岐

エラーに「種類」を持たせて、種類ごとに処理を変える。

例外にも種類があります。計算で起きたなら CalculationError、ユーザー登録で入力が不正なら ValidationError……。種類がまとまっていれば、同じ種類には同じ処理をした方が効率的です（例：入力エラーなら「再入力を促す」）。でも素の Error クラスだけだと、どれがどの種類か区別できません。そこで「自分でエラーの種類を定義」します。

書き方は、Error クラスを継承した新しいクラスを作るだけ。constructor でメッセージを受け取り super(message) で親（Error）に渡し、this.name に独自の名前を入れる——これがカスタムエラーの定番形で、最小単位です。this.name を設定しておくと、エラー表示にその名前（CalculationError）が出ます。使うときは throw new Error(...) を throw new CalculationError(...) に変えるだけ。

種類ごとに処理を分けるのは catch の中でやります。error instanceof CalculationError で「その例外が CalculationError のインスタンスか」を判定します（instanceof は、インスタンスかどうかを調べる演算子）。CalculationError なら「計算のやり直し」のような具体的な処理へ。ここで else を書かないと、それ以外のエラーが来たとき何も起きず「何が起こったか分からない」状態になってしまうので、else は必ず書きます。

POINT
Error を継承してカスタムエラーを作る（super(message) ＋ this.name の定番形）。catch の中で instanceof で種類を判定し、種類ごとに処理を分岐する。想定外に備えて else を必ず書く。
Chapter 8-d

例外処理を使う実例

ユーザー登録を例に、「どこで投げ、どこで受けるか」。

最後に、例外処理が実際どこで使われるかを、ユーザー登録を例に見ます。主役は onSubmit——登録画面で名前と年齢を入力し、確定したときに動く関数のイメージです。onSubmit は受け取った入力をそのまま registerUser に渡します。

registerUser（奥のロジック）では、まず name を trim（前後の空白を削除）し、空なら ValidationError を投げる。続けて age を Number に変換し、有効な数値でない or 負の数なら ValidationError を投げる。両方のガードをくぐり抜けたときだけ「登録しました」に進みます。

ここで大事な考え方が「ガード節」。不正な入力＝もう失敗が確定した状態です。名前が空なのに年齢を検証しても意味がない。だから、失敗が確定した時点でさっさと例外を投げて早めに脱出させ、後続の処理は実行しない——これが安全なコードの基本です。

投げた例外を受け止める（エラーハンドリングする）のは、画面に近い onSubmit 側です。catch した error（unknown 型）が ValidationError のインスタンスなら、そのメッセージを画面に出す想定（ここでは console.error）。一方それ以外の「想定外のエラー」は、扱いを分けます。想定外のエラーは中身が読めない（外部ライブラリの英文メッセージなど）ので、それをそのまま画面に出すのは怖いし、ユーザーフレンドリーでもなく、機密情報が混じる恐れもある。だから「自分で制御できている ValidationError」と「想定外のエラー」で処理を分けるわけです。

POINT
想定できるエラーはカスタムエラーで種別を与え、画面に近い側（境界）で catch して具体的に処理する。想定外のエラーとは扱いを分ける。失敗が確定したら早めに throw して脱出（ガード節）。この設計は UX に直結し、うまくできているほど良いプログラム。
